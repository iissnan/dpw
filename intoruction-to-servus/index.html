Intoruction to Servus

轮子制造的动机

随着项目的开发，项目的文件也越来越多，DPW现在使用的 Grunt 开发方案反应越来越变得迟钝。DPW 的前端构建包含两个阶段，第一个阶段是将两个站点共享的内容与站点本身的内容做一次合并到 Stage 目录；第二阶段在 Stage 目录将进行构建并生成文件到服务器目录下。两个阶段皆需要 Watcher: 第一阶段的源码变动将会触发一次复制到 Stage，而第二阶段变动将会再次执行构建。

这个构建过程的主要问题体现在两个方面：

  • 第一阶段的复制合并需要的时间也越来越长
  • 需要 Watch 的文件越来越多，源码文件的一次改动将会致使第二阶段的 Watcher 重新扫描上千个的文件

方案调研

Gulp 使用虚拟文件（Vinyl Objects）结合 Stream 来减少磁盘的 IO，这在一定程度上可以提升构建的速度，但并没有带来太大的收益，因此使用 Gulp 替代 Grunt 并没有真正的解决问题。

在此期间，曾考虑的使用 Webpack 的 Dev Server 来做 Proxy，通过 Middlewares 处理各个阶段特定的任务。但新版本将使用 Webpack 来做构建，这也就意味着可能需要两个 Webpack Dev Server；另一方面，基于 Grunt 的使用经验，过度依赖一个工具将致使迁移成本很高。比如说，如果现有的 Grunt 任务中，文件的处理采用 NodeJS 来处理而非 Grunt 提供的 API 去处理，那么迁移到 Gulp 的过程将会顺利很多。

综合上述考虑，新的开发工具需要具备以下特性：

  • 工具无关性，将所有文件处理交有 NodeJS 处理。工具将仅根据需要调用这些处理过程。
  • 去除复制到 Stage 的步骤
  • 去除所有的 Watcher
  • 易于扩展，例如支持 Wepack

Servus 登场

于是就有了 Servus。Servus 是一个独立的 Application Server，serve 静态文件和反向代理后端请求，非常类似于 Nginx 静态文件服务器。

Servus 由一个基于 Hapi 编写的 Server 和 Server 启动之前的预处理器组成。Server 通过路由配置，指定特定的请求如何被处理。例如，请求静态脚本文件 asserts/js/login.js 时，此路由对应的处理器会在项目下查找此文件。这种映射请求到磁盘文件的方式，解决了“去除复制到Stage 的复制操作”；同时每一次请求都是直接读取磁盘文件，Watcher 也不再需要。

查找匹配文件的过程速度非常快（通常在 20ms 以下），因此 Servus 带来巨大的速度上的提升。之前 Grunt 需要几分钟的启动过程，Servus 仅需五秒以内（Sass 的预处理花费了一秒多）即可完成；去除 Watcher 后再也不用反复确认文件是否被正确更新，只要保证文件更新有被保存就可以了。

Proxy 部分（主要是 /api/*）使用了穿透模式，也就是客户端的请求和服务器的响应都原封不动直接传递。
